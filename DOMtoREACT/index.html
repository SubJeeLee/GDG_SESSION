4. DOM은 어떻게 생겼을까?

HTML의 포함 관계를 분석해 트리(Tree) 자료구조로 변환합니다.

Node (노드)

DOM 트리를 구성하는 모든 요소를 노드라고 부릅니다. 태그, 텍스트, 속성까지 모두 객체로 관리됩니다.

  • Document Node: 최상단 루트
  • Element Node: <html>, <div> 등
  • Attribute Node: class="box" 등
  • Text Node: 태그 안의 텍스트

6. 순수 DOM 조작의 치명적 함정

Reflow (리플로우)

요소의 너비나 위치가 변하면, 브라우저는 주변 요소들까지 싹 다 위치를 재계산해야 합니다. 이는 매우 무거운 연산입니다.

"만약 바닐라 JS로 리스트 아이템 100개를 하나씩 추가한다면?
브라우저는 화면 레이아웃을 100번 재계산하며 화면이 멈추거나 버벅거리게 됩니다."

7. 개발 패러다임의 거대한 전환

명령형 (바닐라 JS)

"어떻게(How) 바꿀지 절차를 하나하나 지시" 

btn.addEventListener('click', () => {
  msg.innerText = '클릭됨!';
  msg.style.color = 'blue';
});

선언형 (React)

"최종적으로 무엇을(What) 보여줄지 선언" 

return (
  <button onClick={() => setClicked(true)}>
    {clicked ? '클릭됨!' : '클릭전'}
  </button>
);

9. 합리적인 의심: "어차피 DOM 만드는 건 똑같은데?" 🤔

"가상 DOM도 구조를 새로 만드는 거잖아요?
그럼 기존 DOM 수정하는 거랑 똑같이 성능 깎아먹고 느린 거 아닌가요?"

🛋️ 진짜 DOM = 무거운 실제 가구

방 배치를 바꿀 때마다 무거운 진짜 가구를 직접 옮기는 것과 같습니다.

  • 가구 하나를 옮길 때마다 힘이 들고, 다른 가구들도 밀려납니다.
  • 건드리는 순간 무거운 '리플로우(Reflow)' 파이프라인 가동

📄 가상 DOM = 종이 미니어처 도면

진짜 가구를 옮기기 전, 도면 위에서 종이 미니어처를 이리저리 움직여보는 것입니다.

  • 종이를 움직이는 건 힘이 전혀 들지 않습니다 (메모리 상의 객체 조작).
  • 미니어처로 최적의 배치를 다 찾은 뒤, 마지막에 딱 한 번만 진짜 가구를 옮깁니다.

10. "React가 다 해주는데, 왜 DOM을 알아야 할까요?"

🛠️

프레임워크의 한계

특정 위치로 부드럽게 스크롤을 이동하거나(scrollIntoView), 화면 요소의 좌표를 정밀 계산할 때는 결국 순수 DOM API가 필요합니다.

📦

독립적인 라이브러리 제작

React에 종속되지 않는 범용적인 UI 컴포넌트나 라이브러리를 바닥부터 설계하려면 바닐라 JS와 DOM 제어 능력이 필수입니다.

🔍

트러블슈팅의 본질

React 역시 내부적으로는 DOM을 조작하는 도구입니다. 복잡한 렌더링 버그나 성능 이슈를 해결하려면, 그 근간인 DOM의 원리를 알아야만 합니다.

마무리하며

1.DOM은 단순한 HTML 문자열을 자바스크립트가 다룰 수 있게 만든 객체 인터페이스입니다.

2.DOM을 수정할 때마다 발생하는 브라우저의 무거운 레이아웃 재계산을 Reflow라고 합니다.

3.이 Reflow 비용을 최적화하고 생산성을 높이기 위해 탄생한 것이 바로 React와 Virtual DOM입니다.
